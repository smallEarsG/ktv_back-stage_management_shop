# KTV用户端微信小程序 PRD（一期）& 技术架构

## 0. 范围与版本

### 0.1 一期（MVP）目标

* **目标平台**：仅微信小程序。

* **核心闭环**：扫码绑定包厢（房间）-> 点单 -> 微信支付 -> 后台接单/配送。

* **账号体系**：一期需要注册/登录（微信授权登录），用于订单归属与消息触达；不绑定会员体系。

* **数据打通**：复用商户后台的商品/分类/库存/订单/房间等数据。

### 0.2 二期（暂不做）

* 会员体系（会员价、会员卡、积分、储值）

* 优惠券体系

## 1. 产品概述

**产品名称**：KTV自助点单微信小程序

**核心价值**：为到店顾客提供“扫码-点单-支付”的闭环自助服务，减轻服务员人工点单压力，提升下单效率与准确率。

**关联系统**：数据与“KTV商户后台管理系统”完全打通，复用商品、订单、房间等核心数据。

## 2. 用户与流程

### 2.1 用户角色

* **注册用户（一期）**：微信授权登录后点单，下单与订单查询均需登录；不享受会员权益。

* **会员用户（二期）**：在一期登录基础上，开通会员权益与会员价。

### 2.2 核心流程（一期）

1. **进店/入包厢**：顾客进入包厢。
2. **扫码**：扫描桌上/墙上的二维码（房间码）。
3. **绑定房间**：小程序获取 `store_id` 和 `room_id`，进入点单页。
4. **浏览点单**：按分类浏览商品，选择规格，加入购物车。
5. **支付下单**：确认订单 -> 调起微信支付 -> 支付成功。
6. **出品配送**：商户后台收到新订单提醒 -> 服务员接单 -> 配送到房。
7. **订单完成**：用户查看订单状态（待接单 -> 配送中 -> 已完成）。

```mermaid
graph LR
A[用户扫码] --> B{识别房间码}
B -->|无效码| C[提示错误]
B -->|有效码| D[进入点单页(绑定房间)]
D --> E[选购商品]
E --> F[购物车结算]
F --> G[微信支付]
G --> H[生成订单(已支付)]
H --> I[商户后台接单/配送]
```

### 2.3 房间二维码（怎么生成）

* **推荐方式**：生成“微信小程序码”（`getwxacodeunlimit`），扫码直接打开小程序。

* **承载参数**：用 `scene` 携带房间信息。

  * 方案A（短参数）：`scene = "s=12&r=305"`（注意 `scene` 长度限制 32 可见字符）

  * 方案B（推荐，防篡改）：`scene = "t=Ab3K9x"`，小程序启动后调用后端解析 `token -> store_id/room_id`

* **生成时机**：商户后台创建房间后生成/刷新二维码，打印张贴。

## 3. 功能模块（一期）

### 3.1 首页/点单页

* 顶部：门店名称 + 当前房间号（如：V01 包厢）。

* 公告：滚动播放门店公告/促销信息（一期可简化为静态/后台配置）。

* 分类：复用后台商品分类。

* 商品列表：商品图、名称、价格（一期：统一价）。

* 多规格：支持选择规格组合后加入购物车。

* 库存校验：以下单接口为准（前端可做展示与弱校验，防止误导）。

### 3.2 购物车

* 商品汇总：列表展示（一期可不做按分类折叠）。

* 数量调整：增减数量。

* 备注：整单备注。

* 金额计算：合计金额（一期可不做复杂优惠）。

### 3.3 订单中心

* 当前订单：显示当前未完成订单。

* 历史订单：查看历史消费记录。

* 状态：已下单 -> 制作中 -> 配送中 -> 已完成。

* 催单（二期/可选）：若做需限频。

### 3.4 个人中心

* 一期：展示基础入口（例如：我的订单、呼叫服务）。

* 会员卡（二期）：展示会员等级、余额、积分；支持会员价。

### 3.5 呼叫服务

* 一期可提供“呼叫服务/呼叫结账”等按钮，调用后端推送到后台。

## 4. 接口与数据交互（对接现有后端）

### 4.1 API 规划（Prefix：`/api/miniapp`）

| 模块 | 接口路径                     | 方法   | 描述                                           |
| -- | ------------------------ | ---- | -------------------------------------------- |
| 认证 | `/auth/login`            | POST | 微信 `jscode2session` 登录（一期必需：用于下单/查单）         |
| 门店 | `/store/info`            | GET  | 获取门店基础信息（Logo、电话、公告）                         |
| 商品 | `/products`              | GET  | 获取分类及商品列表（仅上架商品）                             |
| 商品 | `/products/{id}`         | GET  | 商品详情（含规格/库存信息）                               |
| 房间 | `/rooms/resolve`         | GET  | `scene` 解析成 `store_id/room_id`（若采用 token 方案） |
| 订单 | `/orders`                | POST | 创建订单（强校验库存、房间状态、价格）                          |
| 订单 | `/orders`                | GET  | 获取我的订单列表                                     |
| 订单 | `/orders/{id}`           | GET  | 订单详情                                         |
| 支付 | `/payment/wechat/prepay` | POST | 获取微信支付预支付参数                                  |
| 服务 | `/service/call`          | POST | 发起呼叫服务（推送消息给后台）                              |

### 4.2 数据模型调整建议

* **房间码解析策略**

  * 方案A：`scene` 直接包含 `store_id/room_id`（短且可读）

  * 方案B：`scene` 存放短 token，后端解析 token（推荐，防篡改）

* **购物车策略（一致性：微信小程序原生）**

  * 前端本地存储：`Storage` 持久化购物车与绑定房间信息

  * 下单时一次性提交明细，后端校验并创建订单

* **商品多规格（已在后台扩展）**

* **C 端用户（一期需要）**

  * 建议新增 `c_users`（或复用现表但区分端）：`openid`（唯一）、`unionid`（可选）、`nickname`、`avatar`、`phone`（可选）、`created_at`、`updated_at`

  * 仅用于登录态与订单归属；不承载会员余额/积分等营销字段（留到二期）

## 5. 技术架构（仅微信小程序）

### 5.1 前端技术栈

* 框架：微信小程序原生（WXML/WXSS/JS）。

* UI：原生组件 + 自研样式（按需封装基础组件）。

* 状态：全局 App State + Storage（购物车、房间绑定、用户信息）。

### 5.2 交互设计风格

* 暗色主题（可选）：考虑包厢光线。

* 大触控区：减少误触，缩短操作链路。

### 5.3 部署架构

* 小程序端：上传至微信公众平台。

* API 服务：复用商户后台 Spring Boot，通过单独的 `/api/miniapp` 路由与鉴权链处理 C 端请求。
